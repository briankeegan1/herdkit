## Design

Six comparable tasks — conformance "none-yet" proof rows (write the missing unit test,
flip the row): HERD-378…380 → **Arm A**, HERD-381…383 → **Arm B**. Same builder model
(claude-sonnet-5) both arms; reviewers claude-opus-4-8; both arms' PRs merged through the
same watcher gate (common quality bar). Token-budget-first per the item's guardrail: no
real quota exhaustion anywhere.

- **Arm A** — herdkit quick lanes (`herd-quick.sh`), the shipped fleet machinery end-to-end.
- **Arm B** — a ~40-line native Claude Code Workflow: `parallel()` fan-out of
  `isolation: 'worktree'` agents, each instructed to build, healthcheck, push, `gh pr create`.
- **Moat leg** — `sandbox-limit-resume-scenario.sh` (drives the *shipped* watcher code):
  sentinel inject → park → scheduled resume (buffer honored) → exactly one `claude` resume.
  The "native arm cannot auto-resume" half is by construction (unshipped CC feature,
  issues #35744/#36320) — cited, not burned tokens on.

## Verdict (the falsification question, answered)

herdkit does **not** justify itself on raw token cost: when everything goes right and a
human is watching, the native workflow delivers the same merged work for comparable or
fewer tokens.

herdkit **does** justify itself on **unattended autonomy**. Every Arm A hiccup was
absorbed by a rail (gate serialization, review pass, reap); every Arm B hiccup converted
directly into operator minutes — and the API-529 death mid-run is the production shape of
the limit-park class the moat targets: work survives only if it happened to cross a
checkpoint before the failure, because nothing notices, retries, or resumes.

Sharpened moat statement (updates the 2026-07-08 assessment): **the moat is the rails plus
the measurement surface** — limit-park auto-resume, refix-with-wake-verification, conflict
resolver, PR adoption, journal/cost ledgers. Recommendation: attended small batches →
native workflow; overnight, fleet, or budget-governed operation → herdkit.
