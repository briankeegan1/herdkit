## Reliability / safety

- тЬЕ **[HIGH] Clean limit-menu resume via herdr pane send-keys (builders + coordinator)** *(PR #124 ┬╖ worktree send-keys-limit-resume)* тАФ **Finding (2026-07-04, operator-driven):** `herdr` has a `herdr pane send-keys <pane_id> <key> [key...]` key-send primitive that the engine's limit-resume has NOT been using. When a session (builder OR coordinator) hits the Claude usage limit it parks at an interactive arrow-menu (`1. Upgrade your plan / 2. Stop and wait for limit to reset`, option 1 highlighted). The clean resume is to send `Down Enter` to select option 2, which triggers Claude Code's OWN native wait-and-auto-resume тАФ the session waits out the limit and continues with full context/work intact. **This REPLACES the current fragile approaches:** (a) re-spawn (loses in-progress uncommitted work); (b) the watcher's `_resume_builder` / coordinator-watchdog firing `claude --continue` at the parsed reset time тАФ this fights the still-menu-parked pane AND depends on a possibly-misparsed banner reset time (observed 2026-07-04: two builders hitting the same limit ~seconds apart got reset targets 7h apart тАФ the reset-time parse is unreliable; file as a sub-bug below). **`herdr send-keys` IS the mechanism, not a contingency (claude-code-guide verdict, 2026-07-04):** there is NO native Claude Code flag/env-var/setting to auto-wait-and-resume on a usage limit тАФ confirmed gap, feature-requested and closed-as-duplicate 3├Ч (anthropics/claude-code issues #26775, #36320, #38263); the 1/2 menu appears even in `-p` / `--dangerously-skip-permissions` mode; `claude --continue` has no documented semantics for a menu-parked session. `herdr pane send-keys <pane> Down Enter` is therefore THE mechanism тАФ and it is strictly BETTER than Claude Code's own documented fallback (a bash re-run loop that loses conversation context). **Remove** the 'if a native flag exists prefer it' branch from the implementation spec тАФ it does not exist. **Proposed:** on detecting a limit-menu-parked session (the `rate_limit` sentinel is already written), send `herdr pane send-keys <pane> Down Enter` to select 'wait for reset' and let Claude's native auto-resume handle it тАФ for BUILDERS (`agent-watch.sh` limit path) AND the COORDINATOR (the coordinator-watchdog, currently default-OFF). **Live verification (2026-07-04 fleet-room smoke test):** `herdr pane send-keys` is PROVEN to deliver keystrokes to live Claude Code TUI dialogs тАФ `Down Enter` successfully navigated and confirmed selections on two real interactive dialogs (permission-approval menus) in the fleet-coordinator pane; `Escape` dismissed an option menu. Key vocabulary (`Down`/`Enter`/`Escape`) and the delivery mechanism are now CONFIRMED live; confidence is high that the shipped send-keys limit-resume (PR #124) will work as designed. **Still to verify LIVE:** (2) confirm that selecting option 2 truly WAITS-AND-AUTO-CONTINUES vs merely stops тАФ live-test on the next real limit-park before treating auto-resume as fully reliable. **Future:** file a Claude Code `/feedback` feature request referencing issues #26775, #36320, #38263 for a native `--auto-resume-on-limit` flag. **Background (2026-06-15):** non-interactive usage (Agent SDK / `claude -p` / GitHub Actions) moved to separate billing tiers тАФ interactive vs non-interactive limits may differ; relevant for overnight-benchmark budgeting. **Sub-bug:** reset-time misparse тАФ two builders hit the same account-wide limit ~seconds apart but the banner-parsed reset time differed by ~7h; the `_resume_builder` timer logic inherited this and fired at the wrong time; the send-keys approach bypasses this entirely (Claude's own timer drives the wait). **Cross-refs:** `limit-resume-menu-hazard` + `unattended-overnight-posture` memories, the coordinator-watchdog item (COORDINATOR auto-resume PR #119), and the herdkit-vs-harness overnight benchmark (this is a prerequisite for a trustworthy unattended run).

- тЬЕ **COORDINATOR auto-resume + watchdog тАФ close the session-scoped single point of failure that breaks unattended overnight runs** *(PR #119)* тАФ **Scope of THIS increment (ЁЯЪз):** RESUME-ONLY half тАФ the COORDINATOR-WATCHDOG piece (piece 1) folded into `agent-watch.sh`, gated behind a **default-OFF `COORDINATOR_WATCHDOG` flag** so the merge is inert; activation = opt-in flag flip + watcher restart, a deliberate operator step. Ships dormant. The EXTERNALIZED WORK-QUEUE half (piece 2) remains ЁЯФЬ as a follow-up; see proposed design (2) below. Design call made autonomously (operator was out for the night, delegated decisions); coordinator did NOT auto-activate it during the build тАФ a fresh unattended coordinator-relauncher is riskier than a clean limit-pause. **THE GAP (confirmed 2026-07-03):** builders get a `rate_limit` `StopFailure` hook (via `new-feature.sh` тЖТ `herd_write_ratelimit_hook`) and the watcher auto-resumes them at reset (`_detect_limit_hit` / `_resume_builder` / `.agent-watch-limit`). The coordinator (`coordinator.sh` тЖТ plain `claude --model MODEL_COORDINATOR /coordinator`) gets NO rate_limit hook, NO watchdog, NO auto-resume; the watcher self-excludes the coordinator tab and never touches it. So if the coordinator hits the account usage limit (or its process dies) unattended, it parks at Claude's interactive '1. Upgrade / 2. Stop and wait' menu and nothing revives it тАФ self-resume paradox: a paused session can't resume itself; only a detached process can. **Impact:** in-flight work still completes (watcher merges ready PRs + resumes builders), but ALL orchestration requiring the coordinator (new spawn waves, human-verify approvals, backlog reaping, escalation handling) halts until a human resumes it тАФ which breaks the 'unattended overnight, survive N limit resets' claim of the herdkit-vs-harness EPIC; the coordinator is the remaining session-scoped seam after execution + state were externalized. **Proposed design тАФ two complementary pieces:** (1) **COORDINATOR-WATCHDOG** тАФ a detached, non-LLM process (sibling to the watcher, or folded into `agent-watch.sh`, or a launchd/cron job) that detects the coordinator is limit-parked or dead (install the `rate_limit` hook on the coordinator like builders have; watch its `.herd-limit-sentinel` + `herdr agent list` status + the interactive menu) and at reset+buffer relaunches it via `claude --continue` in the coordinator pane тАФ mirroring `_resume_builder`; must avoid double-launch (lock like the watcher's `WORKSPACE_NAME` launcher lock), bound retries, and handle the interactive-menu-parked case distinct from a clean session end; ties to the 'validate+harden limit-auto-resume through the interactive menu' item. (2) **EXTERNALIZED WORK QUEUE** тАФ coordinator writes intended builder spawns into a durable queue file that the watcher drains (bounded by `REVIEW_CONCURRENCY`/`SPAWN_AHEAD`), so mechanical throughput survives a coordinator pause; design principle: push everything deterministic into the detached watcher so a coordinator pause barely matters. **Cross-refs:** fleet EPIC's WATCHER HEARTBEAT MONITOR (feature f) and GLOBAL CONCURRENCY/TOKEN GOVERNANCE (feature c); the limit-auto-resume item; the herdkit-vs-harness overnight benchmark (this is a PREREQUISITE for that benchmark to be trustworthy). **Note (2026-07-03):** the earlier live limit-hit did NOT cleanly validate builder auto-resume тАФ a human interfered with the menu before auto-resume could act (see `limit-resume-menu-hazard` memory); both builder-menu-resume AND coordinator-resume need real unattended validation before the overnight benchmark is run.
