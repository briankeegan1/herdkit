## Enterprise / multi-user optionality (deferred — solo is the default today; bank the config seams now)

Reclassify/remove via `herd deps rm|demote` — a dep is editable data, never stuck. *(Gap 3 in `docs/gap-report-cross-repo-loop.md`: `.herd/deps` schema + `herd depend/deps list/rm`; depends on the link registry above for `--to <project>`.)*
- 🔜 **Watcher flexibility for long-pending deps** — backoff polling; richer dep states (`open/in-review/in-progress/stalled/closed`); surface `stalled` + optional TTL so a slow enterprise PR never silently rots. A `blocked-on` is a status line, never a workspace freeze.

- 🔜 **Multi-user / team mode** — `WATCHER_SCOPE=mine|all` + ownership/assignee filter; auto-merge scoped to OWNED PRs only (never blind-merge teammates'), building on the required-checks gate (PR #5). `solo` default; `team` is a config flip.

## Workflow control & discoverability

> Build order: F1 (merge policy) first; capabilities manifest + template-sync guard as one PR; init detection + flow-preference interview as one PR; herd config last (leans on the manifest for key validation).

- 🔜 **Flow-preference interview + draft-PR flow** — New `herd init` questions → config keys: `PR_FLOW=direct|draft` (draft: lanes instruct builders `gh pr create --draft`; watcher already holds `DRAFT`, agent-watch.sh:157), `PR_READY_WHEN=builder|coordinator|human`, `LOCAL_REVIEW=none|pre-pr` (run the review gate in the worktree BEFORE the PR is public vs today's post-PR review), `MERGE_METHOD`, `DELETE_BRANCH_ON_MERGE`; thread prefs into the lane rules text (herd-quick.sh:58-61, herd-feature.sh:50-53).
- 🔜 **`herd config` + coordinator Workflow settings mode** — `herd config list|get|set` with key validation, aware of what each change requires: watcher keys ⇒ restart watcher (pid file `.watcher-<slug>.pid`), coordinator-facing keys ⇒ re-render skill (render step at bin/herd:277-278); coordinator menu gains a 'Workflow settings' entry to view/change any workflow pref anytime and relaunch affected pieces — nothing is init-only.
- 🔜 **Backlog-reconcile step in cutover / extraction / rename PRs** *(issue #17)* — when a PR moves or renames the things that backlog items point at (file paths, function names, section headers, worktree names), affected entries dangle silently after the PR lands. The coordinator / scribe should run a reconcile pass as part of landing such PRs: diff the rename surface, identify backlog entries that reference moved things, and update or flag them so the backlog stays coherent with the live codebase.

## Reliability / safety

- 🔜 **Layout reconciler — engine must observe the live pane layout before mutating it (eyes-on-layout)** *(user insight 2026-07-02)* — every pane-mutating path (`cmd_reload`, `coordinator.sh`, future `herd pane` subcommands) acts on beliefs — the `.herd-panes` registry (stale/poisonable, e.g. Path B overwrote it with stray-tab IDs) and one-shot geometric neighbor guesses — instead of observing reality; a herd reload built standalone tabs and a rerun split a duplicate backlog-view pane. The engine already has the eyes: `herdr pane list` + `herdr pane process-info` can classify every pane in a tab by the script it runs (`backlog-view.sh`, `agent-watch.sh`/`herd-watch.sh`, claude agent, bare shell). Build a shared layout-snapshot helper in `scripts/herd` (scan tab → roles → reconcile against desired layout: adopt panes already serving a role, create only what is missing, close/flag duplicates, rewrite the registry from the observed scan on every mutation) and make `cmd_reload` + `coordinator.sh` consume it; hermetic tests with faked pane/process-info JSON covering stale-registry, duplicate-viewer, and missing-pane cases. Complements the in-flight reload-anchor-no-agent fix (a scan-before-mutate addendum was sent to that builder) and the sandbox-consumer sim item, which verifies this visually. Sequence after `reload-anchor-no-agent` merges. **Geometry mechanics (learned 2026-07-02 live-layout fix):** (1) Desired-geometry spec: backlog viewer occupies the full-height left column; right column = coordinator pane over watch pane, split at ratio 0.72 (`coordinator.sh:84`). (2) `herdr pane move <pane> --tab <t> --split down --target-pane <p> --ratio N` re-parents a live pane without killing its process, but a same-tab move is a no-op (returns `changed:false reason:same_tab`) — the working recipe is bounce out via `--new-tab` then move back in with `--target-pane`; the temp tab auto-closes when emptied. (3) Width-sensitive viewers (`backlog-view.sh` re-renders only on mtime/banner change) need a repaint nudge after geometry changes — `touch`-ing `BACKLOG.md` suffices since the render loop reads `tput cols` per frame.
- 🔜 **Watcher process must carry a per-workspace argv0 so cross-project sweeps cannot kill the wrong instance** — every project sharing one herdkit install runs the watcher via the identical script path (`bash <herdkit>/scripts/herd/agent-watch.sh`), so any kill-by-pattern — a builder's test sweep, a user's `pkill -f agent-watch.sh`, or a `cmd_reload` — hits EVERY project's watcher at once; observed 2026-07-02: northstar's watcher was found dead (no process, no lockfile) as apparent collateral from herdkit-side sweeps. Fix: `agent-watch.sh` should `exec -a herd-watch-<WORKSPACE_NAME>` itself at startup so the process appears in listings under a distinctive per-workspace name; `cmd_reload` and any `pgrep` guidance in docs/lane-rules should match `herd-watch-<WORKSPACE_NAME>` rather than the shared script path; this also makes orphan detection more reliable since a live watcher's name encodes its workspace.
- 🔜 **Launch-binding guard — engine scripts must refuse to run against the wrong project** — `herd-config.sh` resolves config by env → `$PWD`-walk-up → fallback to the engine repo's own dogfood config (rule 3, `herd-config.sh:35`); that fallback silently binds a console launched from a non-project directory (e.g. `$HOME`) to HERDKIT's config — observed 2026-07-02: the user's 'northstar watcher' pane was actually a herdkit-config watcher all along (same lockfile), so killing herdkit's watcher pid appeared to kill 'the other repo's watcher'; northstar-trees had never even had a pid file. Fix: (a) long-running consoles (`agent-watch`/`herd-watch`, `backlog-view`, `coordinator.sh`) must print a startup banner with the resolved `WORKSPACE_NAME` + `PROJECT_ROOT` and REFUSE to start when `$PWD` is not inside the resolved `PROJECT_ROOT` (override flag for intentional cases); (b) consider dropping fallback rule 3 for consoles entirely — walk-up-or-die; (c) per-workspace `argv0` naming (existing backlog item) complements this so `ps` output is unambiguous.
- 🔜 **Auto-refix records a bounce without verifying the builder woke** — first live firing (PR #43 BLOCK, 2026-07-02 ~13:06Z): the watcher wrote the sha-keyed row to `.agent-watch-refixed`, but the builder agent never flipped to working and no fix was pushed — the agent's status was 'done' (finished session), and the bounce evidently didn't submit or the verify-flip check didn't run/failed silently; the spec required verify agent_status→working within ~15s, one retry, then a loud 'needs you · auto-refix failed' console escalation — none of that surfaced. Investigate agent-watch.sh's refix delivery: does it verify the flip at all; does it handle the 'done' (session-ended) agent state — an ENDED claude session may need a fresh 'claude -c'/resume or a new agent start in the pane rather than typed text; assert the escalation path fires when the wake fails. Extend tests with an agent that stays 'done' after the bounce.
- 🔜 **Auto-resume a builder that hit the usage limit (detect → wait for reset → `claude --continue`) instead of freezing or needing a manual respawn** *(incident 2026-07-02: the #60 workspace-scoped-reload builder AND the scribe both hit the session limit mid-task — banner "You have hit your session limit - resets 7:30pm" — and froze with `herdr status done/blocked`; typed nudges after quota reset were ignored, so the coordinator had to kill + respawn, losing the in-progress uncommitted worktree edits)* — **Design fork (resolve first):** dispatch a `claude-code-guide` investigation: does the Claude Code CLI already have a native "wait for reset and auto-continue" mode? If yes, launch builders to use that mode and have the watcher merely suppress false-kill and show the right status row; if no, build the engine-managed resume below. **Detection hook:** `_classify_builder` (agent-watch.sh:975) already reads transcript files via `_transcript_growing` (line 929); extend it to also inspect the last assistant message in the transcript for the limit banner (or, preferably, watch for a structured exit-code signal from the CLI if one is exposed — regex on banner text is version-brittle); when detected, classify the builder as `LIMIT_HIT` rather than `STALL` or `done`. Parse the reset time from the banner (or use a configurable default offset). **Console surface:** emit a dedicated row — `limit-hit · auto-resume at <HH:MM>` — styled distinctly from stall/red rows (NOT a red alarm, NOT an infra transient); journal via `journal_append limit_detected reset_at <epoch>` and `journal_append limit_resume_scheduled target_pane <id>` (following the pattern of `refix_bounce` / `refix_wake_result` at agent-watch.sh:791,819). **Resume mechanic:** at reset + a small buffer, run `claude --dangerously-skip-permissions --continue` IN THE BUILDER'S WORKTREE — this preserves full session context; typed text via `herdr pane run` (the current refix delivery at agent-watch.sh:802,808) does NOT wake a limit-blocked session and MUST NOT be used here. After the relaunch, reuse `_wait_agent_working` (agent-watch.sh:742) to poll `agent_status` for the flip to `"working"`; retry once; escalate loudly as `"needs you · limit-resume failed"` on failure, mirroring the refix escalation path. **Shared helper:** the resume-builder-in-place sequence (`claude --continue` in the correct worktree, verify wake via `_wait_agent_working`, escalate) is the SAME operation needed by the auto-refix item when the target agent is in `done` state (session ended); extract it as a single `_resume_builder <worktree> <pane_id>` helper in agent-watch.sh consumed by both features — do NOT duplicate. **Hermetic tests:** stub a builder whose transcript ends at the limit banner; assert detection, the limit-row label (not red/stall), the scheduled-resume invocation shape (`claude --continue` in the correct worktree path), the wake-verify polling, and the escalation when the agent stays non-working; never touch live watcher/panes. Sequence after the auto-refix wake-verify fix merges (both share the `_resume_builder` helper).
- 🔜 **Sandbox consumer — a dummy repo + herdr workspace for live workflow simulation, especially pane/tab behavior** — every painful bug on 2026-07-02 was integration-level and VISUAL (review pane blank from `claude -p` buffering; reload's watcher pane sitting at a bare prompt while the process ran detached; spawns landing in whichever workspace had focus; stranded review/resolve tabs; cp1252 silently truncating the control room) — the hermetic test suite structurally cannot see any of these because it stubs herdr and never renders a pane. Build a live simulation rig: (1) a dedicated sandbox repo hosted on GitHub (e.g. `herdkit-sandbox`: a tiny real app + seeded BACKLOG.md) with a `herd sim init`-style fixture generator that deterministically resets it (fresh branch, clean worktrees dir, seeded issues) so every run starts identical — reuse the conventions from the shipped cross-repo loop simulation (PR #24, scripts/herd/sim); (2) a dedicated herdr workspace ('sandbox') where scenarios spin up a REAL control room so pane/tab experiments never touch real projects; assert placement/layout/labels/teardown via `herdr tab list` / `pane list` / `agent list` JSON at each checkpoint; (3) scripted end-to-end scenarios: init → control-room launch → spawn builder → PR → review gate → merge → teardown, with a STUB builder agent (deterministic tiny change, no model call) so runs are fast/cheap, plus an opt-in real-model smoke mode; (4) VISUAL confirmation: capture pane contents (herdr capture API if available, else launch a terminal window and take OS screenshots, e.g. macOS `screencapture` of the herdr window) at checkpoints; keep golden screenshots and have a vision-capable `claude -p` judge assert observable truths ('3 panes present', 'watch console painting rows', 'review pane streaming text') rather than pixel-diffing; (5) fault-injection scenarios reproducing 2026-07-02's bug classes: kill the watcher mid-review (expect INFRA retry not cached BLOCK), never-committed backlog file, `PYTHONIOENCODING=ascii` to simulate cp1252, missing optional deps (glow), duplicate-watcher race, focused-workspace misdirection — assert LOUD failures, never silent truncation; (6) wire as an opt-in heavy gate: engine PRs touching pane-creating scripts (coordinator.sh, lanes, herd-review.sh, agent-watch.sh, cmd_reload) should trigger a sandbox sim run before merge — like `HEALTHCHECK_HEAVY_GLOB` but for workflow surface; (7) each sim run ends with a machine-readable scorecard (scenario, checkpoints passed, screenshots dir) so the coordinator can review results and a human can eyeball the visual evidence.
