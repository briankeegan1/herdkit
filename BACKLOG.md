## Workflow control & discoverability

> Build order: F1 (merge policy) first; capabilities manifest + template-sync guard as one PR; init detection + flow-preference interview as one PR; herd config last (leans on the manifest for key validation).

- 🔜 **Init-time GitHub detection** — `cmd_init` (bin/herd:104-284) makes zero `gh api` calls today; add a detection pass: branch protection on the default branch, required-review count, required checks, CODEOWNERS, allowed merge methods; show findings and derive safe defaults (required reviewers present ⇒ default `MERGE_POLICY=approve`, never `auto`); degrade gracefully without remote/gh.
- 🔜 **Flow-preference interview + draft-PR flow** — New `herd init` questions → config keys: `PR_FLOW=direct|draft` (draft: lanes instruct builders `gh pr create --draft`; watcher already holds `DRAFT`, agent-watch.sh:157), `PR_READY_WHEN=builder|coordinator|human`, `LOCAL_REVIEW=none|pre-pr` (run the review gate in the worktree BEFORE the PR is public vs today's post-PR review), `MERGE_METHOD`, `DELETE_BRANCH_ON_MERGE`; thread prefs into the lane rules text (herd-quick.sh:58-61, herd-feature.sh:50-53).
- 🔜 **`herd config` + coordinator Workflow settings mode** — `herd config list|get|set` with key validation, aware of what each change requires: watcher keys ⇒ restart watcher (pid file `.watcher-<slug>.pid`), coordinator-facing keys ⇒ re-render skill (render step at bin/herd:277-278); coordinator menu gains a 'Workflow settings' entry to view/change any workflow pref anytime and relaunch affected pieces — nothing is init-only.
- 🔜 **Backlog-reconcile step in cutover / extraction / rename PRs** *(issue #17)* — when a PR moves or renames the things that backlog items point at (file paths, function names, section headers, worktree names), affected entries dangle silently after the PR lands. The coordinator / scribe should run a reconcile pass as part of landing such PRs: diff the rename surface, identify backlog entries that reference moved things, and update or flag them so the backlog stays coherent with the live codebase.
- 🔜 **Analyst seat launch config must include an explicit context-persistence tenet** *(issue #19)* — the playground analyst seat's herdkit-hosted launch prompt has no cross-session context-persistence instruction, so the seat can silently go stateless between sessions: the only instance of that behavior today survives as a pruneable memory entry in one consumer project (northstar), not as a durable directive in the seat config. Fix: add an explicit tenet to the analyst seat's launch prompt / config requiring it to actively build and update the user's project and domain context in the local project memory store on every session — context stays local and gitignored per the existing memory contract; this is a prompt/config-only change with no engine wiring.
- 🚧 **`herd pane <watch|backlog|coordinator>` — single-pane restart shortcuts** *(worktree `herd-pane-shortcuts`)* — user request: a one-liner to re-run one specific control-room pane without a full reload (today it's a manual kill-by-lockfile + rerun-in-pane dance). Build on reload-in-place's `.herd-panes` role registry (worktree `reload-in-place`, PR #44): `herd pane watch` stops this project's watcher via its lockfile and reruns `herd-watch.sh` in the registered watch pane (recreating the pane in the coordinator tab if gone); `herd pane backlog` reruns `backlog-view.sh` in the registered backlog pane; `herd pane coordinator` relaunches the coordinator agent pane ONLY on explicit confirmation (it kills the live coordinator session); capabilities.tsv rows; hermetic tests. Sequence after PR #44 merges.
- 🔜 **Per-PR human-verify hold — PR test plans must be visible to the gate** *(user insight on PR #44)* — a PR whose description contains a manual test step (e.g. 'Live smoke: run coordinator.sh, verify .herd-panes, verify reload refreshes panes') sails through the correctness review and AUTO-MERGES with that step silently unexecuted — the pipeline has no concept of human-required verification. Build: (1) a convention builders emit in the PR body — a `HUMAN-VERIFY:` block listing steps they could not perform themselves — plus lane-rules text instructing builders to declare any such steps; (2) the watcher parses open PRs for the marker and switches just that PR to an approve-style hold (reuse the sha-keyed awaiting/approval ledger from `MERGE_POLICY=approve` — gates still run, merge waits) and shows a console row like `ready · human-verify pending · herd-approve.sh approve <pr#>`; (3) the coordinator surfaces held PRs with their verify steps so the human (or the coordinator itself, when steps are safely executable) can run them and approve; (4) once the sandbox-sim rig exists, steps matching automatable patterns route there instead of a human; (5) capabilities.tsv rows + hermetic tests covering marker parsing, per-PR hold while other PRs auto-merge, and approval release.

## Reliability / safety

- 🔜 **Layout reconciler — engine must observe the live pane layout before mutating it (eyes-on-layout)** *(user insight 2026-07-02)* — every pane-mutating path (`cmd_reload`, `coordinator.sh`, future `herd pane` subcommands) acts on beliefs — the `.herd-panes` registry (stale/poisonable, e.g. Path B overwrote it with stray-tab IDs) and one-shot geometric neighbor guesses — instead of observing reality; a herd reload built standalone tabs and a rerun split a duplicate backlog-view pane. The engine already has the eyes: `herdr pane list` + `herdr pane process-info` can classify every pane in a tab by the script it runs (`backlog-view.sh`, `agent-watch.sh`/`herd-watch.sh`, claude agent, bare shell). Build a shared layout-snapshot helper in `scripts/herd` (scan tab → roles → reconcile against desired layout: adopt panes already serving a role, create only what is missing, close/flag duplicates, rewrite the registry from the observed scan on every mutation) and make `cmd_reload` + `coordinator.sh` consume it; hermetic tests with faked pane/process-info JSON covering stale-registry, duplicate-viewer, and missing-pane cases. Complements the in-flight reload-anchor-no-agent fix (a scan-before-mutate addendum was sent to that builder) and the sandbox-consumer sim item, which verifies this visually. Sequence after `reload-anchor-no-agent` merges. **Geometry mechanics (learned 2026-07-02 live-layout fix):** (1) Desired-geometry spec: backlog viewer occupies the full-height left column; right column = coordinator pane over watch pane, split at ratio 0.72 (`coordinator.sh:84`). (2) `herdr pane move <pane> --tab <t> --split down --target-pane <p> --ratio N` re-parents a live pane without killing its process, but a same-tab move is a no-op (returns `changed:false reason:same_tab`) — the working recipe is bounce out via `--new-tab` then move back in with `--target-pane`; the temp tab auto-closes when emptied. (3) Width-sensitive viewers (`backlog-view.sh` re-renders only on mtime/banner change) need a repaint nudge after geometry changes — `touch`-ing `BACKLOG.md` suffices since the render loop reads `tput cols` per frame.

- 🚧 **Stall detector: check worktree mtimes before flagging** *(USER PRIORITY — next agent-watch.sh change after healthcheck-gate-accuracy lands; second false-alarm complaint of 2026-07-02, extends the 'no false reds' directive to yellow stall warnings)* *(worktree: stall-detector-liveness)* — `agent-watch.sh:383-392` flags any working builder with zero branch commits after 5 minutes as 'stalled? · check pane', but builders normally commit ONCE at the end right before `gh pr create`, so every normal build >5 min shows a false stall warning (observed 2026-07-01: herd-reload and workspace-pin both flagged while actively editing — one had file edits 0 minutes old; observed again 2026-07-02: builder actively coding, watcher showed 'stalled? · check pane' prematurely — the watcher doesn't account for long normal coding time before the single end-of-build commit). Refine: before flagging, check the newest mtime among the worktree's dirty/untracked files (`git status --porcelain` paths); recent edits (< ~5 min) ⇒ show 'building (uncommitted changes)' instead of the stall warning; only flag when the tree is clean AND commitless AND old, or dirty but untouched for >N min. Two additional cheap liveness signals to layer in: (a) `herdr` already reports `agent_status=working` straight from the Claude CLI — a working agent with recent worktree edits must ALWAYS render 'building (uncommitted changes)', never 'stalled?'; (b) the Claude session transcript file for the worktree grows while the agent works — a flatlined transcript PLUS untouched tree PLUS zero commits is the genuinely-dead signature that alone justifies the stall warning. Keep the warning for the genuinely-dead case (agent status not 'working' already shows 'idle · no PR').
- 🔜 **Healthcheck must gate / flag interaction tests for app-facing PRs** *(issue #15)* — the consumer healthcheck profile today runs only a render smoke (e.g. `st.testing.v1.AppTest.from_file`), which is blind to broken interactivity: a persistent quick-set radio silently overrode a slider every re-run in northstar PR #48 while the smoke passed. Engine-side ask: the lane rules / healthcheck contract should let a project declare interaction-test expectations (e.g. Streamlit `AppTest`: drive the widget, then assert the dependent output changes) and at minimum flag their *absence* for PRs that touch the app surface — a healthcheck that cannot observe widget → output causality should not silently green-light an app-facing PR.
- 🔜 **Watcher process must carry a per-workspace argv0 so cross-project sweeps cannot kill the wrong instance** — every project sharing one herdkit install runs the watcher via the identical script path (`bash <herdkit>/scripts/herd/agent-watch.sh`), so any kill-by-pattern — a builder's test sweep, a user's `pkill -f agent-watch.sh`, or a `cmd_reload` — hits EVERY project's watcher at once; observed 2026-07-02: northstar's watcher was found dead (no process, no lockfile) as apparent collateral from herdkit-side sweeps. Fix: `agent-watch.sh` should `exec -a herd-watch-<WORKSPACE_NAME>` itself at startup so the process appears in listings under a distinctive per-workspace name; `cmd_reload` and any `pgrep` guidance in docs/lane-rules should match `herd-watch-<WORKSPACE_NAME>` rather than the shared script path; this also makes orphan detection more reliable since a live watcher's name encodes its workspace.
- 🔜 **Launch-binding guard — engine scripts must refuse to run against the wrong project** — `herd-config.sh` resolves config by env → `$PWD`-walk-up → fallback to the engine repo's own dogfood config (rule 3, `herd-config.sh:35`); that fallback silently binds a console launched from a non-project directory (e.g. `$HOME`) to HERDKIT's config — observed 2026-07-02: the user's 'northstar watcher' pane was actually a herdkit-config watcher all along (same lockfile), so killing herdkit's watcher pid appeared to kill 'the other repo's watcher'; northstar-trees had never even had a pid file. Fix: (a) long-running consoles (`agent-watch`/`herd-watch`, `backlog-view`, `coordinator.sh`) must print a startup banner with the resolved `WORKSPACE_NAME` + `PROJECT_ROOT` and REFUSE to start when `$PWD` is not inside the resolved `PROJECT_ROOT` (override flag for intentional cases); (b) consider dropping fallback rule 3 for consoles entirely — walk-up-or-die; (c) per-workspace `argv0` naming (existing backlog item) complements this so `ps` output is unambiguous.
