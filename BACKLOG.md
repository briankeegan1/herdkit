## Reliability / safety

- ✅ **[HIGH] Clean limit-menu resume via herdr pane send-keys (builders + coordinator)** *(PR #124 · worktree send-keys-limit-resume)* — **Finding (2026-07-04, operator-driven):** `herdr` has a `herdr pane send-keys <pane_id> <key> [key...]` key-send primitive that the engine's limit-resume has NOT been using. When a session (builder OR coordinator) hits the Claude usage limit it parks at an interactive arrow-menu (`1. Upgrade your plan / 2. Stop and wait for limit to reset`, option 1 highlighted). The clean resume is to send `Down Enter` to select option 2, which triggers Claude Code's OWN native wait-and-auto-resume — the session waits out the limit and continues with full context/work intact. **This REPLACES the current fragile approaches:** (a) re-spawn (loses in-progress uncommitted work); (b) the watcher's `_resume_builder` / coordinator-watchdog firing `claude --continue` at the parsed reset time — this fights the still-menu-parked pane AND depends on a possibly-misparsed banner reset time (observed 2026-07-04: two builders hitting the same limit ~seconds apart got reset targets 7h apart — the reset-time parse is unreliable; file as a sub-bug below). **`herdr send-keys` IS the mechanism, not a contingency (claude-code-guide verdict, 2026-07-04):** there is NO native Claude Code flag/env-var/setting to auto-wait-and-resume on a usage limit — confirmed gap, feature-requested and closed-as-duplicate 3× (anthropics/claude-code issues #26775, #36320, #38263); the 1/2 menu appears even in `-p` / `--dangerously-skip-permissions` mode; `claude --continue` has no documented semantics for a menu-parked session. `herdr pane send-keys <pane> Down Enter` is therefore THE mechanism — and it is strictly BETTER than Claude Code's own documented fallback (a bash re-run loop that loses conversation context). **Remove** the 'if a native flag exists prefer it' branch from the implementation spec — it does not exist. **Proposed:** on detecting a limit-menu-parked session (the `rate_limit` sentinel is already written), send `herdr pane send-keys <pane> Down Enter` to select 'wait for reset' and let Claude's native auto-resume handle it — for BUILDERS (`agent-watch.sh` limit path) AND the COORDINATOR (the coordinator-watchdog, currently default-OFF). **Still to verify LIVE (docs couldn't confirm):** (1) confirm `herdr send-keys` exact key-name vocabulary (`Down`/`Enter` vs `Return` etc.) and that it delivers the arrow+enter to the TUI; (2) confirm that selecting option 2 truly WAITS-AND-AUTO-CONTINUES vs merely stops — live-test on the next real limit-park before treating auto-resume as reliable. **Future:** file a Claude Code `/feedback` feature request referencing issues #26775, #36320, #38263 for a native `--auto-resume-on-limit` flag. **Background (2026-06-15):** non-interactive usage (Agent SDK / `claude -p` / GitHub Actions) moved to separate billing tiers — interactive vs non-interactive limits may differ; relevant for overnight-benchmark budgeting. **Sub-bug:** reset-time misparse — two builders hit the same account-wide limit ~seconds apart but the banner-parsed reset time differed by ~7h; the `_resume_builder` timer logic inherited this and fired at the wrong time; the send-keys approach bypasses this entirely (Claude's own timer drives the wait). **Cross-refs:** `limit-resume-menu-hazard` + `unattended-overnight-posture` memories, the coordinator-watchdog item (COORDINATOR auto-resume PR #119), and the herdkit-vs-harness overnight benchmark (this is a prerequisite for a trustworthy unattended run).

- ✅ **Dead-builder detection — surface a spawned builder whose agent process vanished** *(PR #117)* — detect the case where a worktree is present, no live agent exists in `herdr agent list`, and no open PR has been filed; emit a loud console row + notification so the operator knows the builder died silently. Motivated by the 2026-07-03 de-streamlit-docs builder silently dying (agent gone, pane destroyed, no PR, no commits) — found only when the operator noticed an empty tab. The watcher already handles (a) transcript-growth STALL for working-but-stuck builders (~line 1352), (b) `_sweep_orphan_tabs` for tabs with no worktree/PR (~line 749), (c) limit-hit auto-resume (~line 132) — the dead-pre-PR case falls through all of them. First increment: DETECT + loud alert only; auto-respawn deferred (edge cases: uncommitted work, respawn loops). Extends existing builder-liveness section without duplicating it. A broader control-room health sweep (watcher + backlog viewer + coordinator panes all alive) is a possible follow-up; cross-refs fleet EPIC's WATCHER HEARTBEAT MONITOR (feature f).
- ✅ **COORDINATOR auto-resume + watchdog — close the session-scoped single point of failure that breaks unattended overnight runs** *(PR #119)* — **Scope of THIS increment (🚧):** RESUME-ONLY half — the COORDINATOR-WATCHDOG piece (piece 1) folded into `agent-watch.sh`, gated behind a **default-OFF `COORDINATOR_WATCHDOG` flag** so the merge is inert; activation = opt-in flag flip + watcher restart, a deliberate operator step. Ships dormant. The EXTERNALIZED WORK-QUEUE half (piece 2) remains 🔜 as a follow-up; see proposed design (2) below. Design call made autonomously (operator was out for the night, delegated decisions); coordinator did NOT auto-activate it during the build — a fresh unattended coordinator-relauncher is riskier than a clean limit-pause. **THE GAP (confirmed 2026-07-03):** builders get a `rate_limit` `StopFailure` hook (via `new-feature.sh` → `herd_write_ratelimit_hook`) and the watcher auto-resumes them at reset (`_detect_limit_hit` / `_resume_builder` / `.agent-watch-limit`). The coordinator (`coordinator.sh` → plain `claude --model MODEL_COORDINATOR /coordinator`) gets NO rate_limit hook, NO watchdog, NO auto-resume; the watcher self-excludes the coordinator tab and never touches it. So if the coordinator hits the account usage limit (or its process dies) unattended, it parks at Claude's interactive '1. Upgrade / 2. Stop and wait' menu and nothing revives it — self-resume paradox: a paused session can't resume itself; only a detached process can. **Impact:** in-flight work still completes (watcher merges ready PRs + resumes builders), but ALL orchestration requiring the coordinator (new spawn waves, human-verify approvals, backlog reaping, escalation handling) halts until a human resumes it — which breaks the 'unattended overnight, survive N limit resets' claim of the herdkit-vs-harness EPIC; the coordinator is the remaining session-scoped seam after execution + state were externalized. **Proposed design — two complementary pieces:** (1) **COORDINATOR-WATCHDOG** — a detached, non-LLM process (sibling to the watcher, or folded into `agent-watch.sh`, or a launchd/cron job) that detects the coordinator is limit-parked or dead (install the `rate_limit` hook on the coordinator like builders have; watch its `.herd-limit-sentinel` + `herdr agent list` status + the interactive menu) and at reset+buffer relaunches it via `claude --continue` in the coordinator pane — mirroring `_resume_builder`; must avoid double-launch (lock like the watcher's `WORKSPACE_NAME` launcher lock), bound retries, and handle the interactive-menu-parked case distinct from a clean session end; ties to the 'validate+harden limit-auto-resume through the interactive menu' item. (2) **EXTERNALIZED WORK QUEUE** — coordinator writes intended builder spawns into a durable queue file that the watcher drains (bounded by `REVIEW_CONCURRENCY`/`SPAWN_AHEAD`), so mechanical throughput survives a coordinator pause; design principle: push everything deterministic into the detached watcher so a coordinator pause barely matters. **Cross-refs:** fleet EPIC's WATCHER HEARTBEAT MONITOR (feature f) and GLOBAL CONCURRENCY/TOKEN GOVERNANCE (feature c); the limit-auto-resume item; the herdkit-vs-harness overnight benchmark (this is a PREREQUISITE for that benchmark to be trustworthy). **Note (2026-07-03):** the earlier live limit-hit did NOT cleanly validate builder auto-resume — a human interfered with the menu before auto-resume could act (see `limit-resume-menu-hazard` memory); both builder-menu-resume AND coordinator-resume need real unattended validation before the overnight benchmark is run.

## Efficiency / cost

> Spin-outs from the deferred [herdkit-vs-harness efficiency EPIC](#someday--deferred) (surfaced 2026-07-02). Token/cost instrumentation — `scripts/herd/cost.sh`, per-builder/review token accounting priced to USD, cost-per-merged-PR, `herd cost` command — is ✅ shipped (commit e2a0ae5); the instrumentation gate is lifted for all items below.


- ✅ **AUDIT (research pass): end-to-end flow redundancy, speed-ups, and gate necessity/correctness** — **Deliverable:** a findings doc (`docs/flow-audit.md`) + a RANKED list of concrete follow-up items, not a single code change; this is the umbrella under which the six spun-out efficiency levers below sit. **Why (surfaced 2026-07-02 to 2026-07-03):** the sha-cache fix (a held PR re-ran the full healthcheck every ~90s tick and manufactured its own transient code-error→flaky on PR #65) is likely one symptom of a broader pattern — the whole pipeline probably repeats work and runs gates that are redundant, unnecessary for some diffs, or sequenced sub-optimally. **Scope — every stage end-to-end:** (1) backlog/scribe — per-item git pulls, drainer spawn/teardown churn; (2) lane spawn — worktree + SHARE_LINKS symlinks, multi-kilobyte argv prompts; (3) build — isolated builders re-deriving context from files; (4) WATCHER TICK LOOP (`agent-watch.sh`) — how many `gh`/`herdr`/`git` subprocesses it forks per tick (latency + rate-limit + cost), what it recomputes every tick vs could cache, and whether the tick cadence itself is right; (5) HEALTHCHECK gate — serialization (`HEALTH_CONCURRENCY=1`: always necessary, or can worktree-isolated suites parallelize?), retry-before-red, and the now-added sha-cache; (6) REVIEW gate — per-PR Opus cost, whether trivial/docs/test-only diffs need it at all (ties to the risk-tiered-review item), and whether review + healthcheck overlap; (7) auto-refix rounds + wake verification; (8) merge re-verification; (9) reap — worktree remove, tab close, backlog reap, orphan sweep. **Look for four things at each stage:** (a) REDUNDANT work (repeated subprocess calls, re-reads, re-runs of unchanged inputs); (b) SPEED-UPS (caching, parallelization, better sequencing, cheaper models/tools); (c) UNNECESSARY gates — anything duplicating GitHub's own branch-protection/required-checks or adding latency without safety value; (d) gate ISSUES / correctness — races, false-reds, the manufactured-transient class, gates that can cache a wrong verdict. **Method:** token/cost instrumentation is ✅ shipped (commit e2a0ae5 — `herd cost`; per-builder/review accounting); add latency instrumentation as the audit proceeds, then map the flow, then rank findings; each significant finding becomes its own item; validate any flow change on the sandbox-consumer sim rig before shipping. **Composes with:** the six efficiency levers below (its children), the herdkit-vs-harness EPIC in `## Someday / Deferred` (shares the instrumentation precursor), and the sandbox-consumer rig (its safe test bed). **DELIVERABLE SHIPPED:** `docs/flow-audit.md` landed as PR #77 (commit 14f7e42); ranked follow-up items from that doc still need triaging into individual backlog items.
- ✅ **Instrument COORDINATOR's own token/cost — `herd cost` is builder+review only** *(worktree coordinator-cost-full, shipped PR #125)* — **Problem:** `cost.sh` (commit e2a0ae5) emits token/USD events only via `cost_emit_merge` — one event per MERGED builder and one per review, both gated on a merge happening. The coordinator's own orchestration turns (large-context Opus reasoning every tick), scribe drainers, and researcher agents are NOT counted, and neither are failed/torn-down/limit-re-spawned builders that never reached a merge. The result is that `herd cost` reports a real but materially incomplete figure: ~$65 measured via `herd cost` / `cost.sh` across 13 PRs in a session where the true all-in cost was ~$300 (rough operator estimate 2026-07-04). **Proposed:** extend `cost.sh` to also read the COORDINATOR's transcript (and optionally scribe/researcher transcripts) and emit cost events for them into the journal, so `herd cost` reports a complete session total (coordinator + builders + reviews + scribes), not just merge-gated builder/review spend. The coordinator tab already has a transcript dir resolvable via the same `_cost_transcript_dir` helper used for builders. **Design nuance — emit trigger:** the coordinator is a long-lived session with NO merge event to hook into, unlike builders; needs either (a) periodic emission among coordinator ticks (e.g. emit a cost snapshot to the journal every N minutes), or (b) an on-demand `herd cost --full` subcommand that scans all live transcripts at call time and aggregates them alongside the existing journal events; decide which model before building — option (b) is simpler and avoids journal churn, option (a) gives real-time cost visibility during a run. **Also scope in:** (c) builders that die before merge (the dead-builder detection item, PR #117, already surfaces them — pair their transcript cost into the report); (d) a `herd cost --session` view that groups by session/date rather than by PR, so the operator can see the real all-in cost of an overnight run, not just the per-PR average. **Cross-refs:** mirrors `cost.sh`'s existing `_cost_transcript_dir` transcript-token parsing; composes with the herdkit-vs-harness EPIC's cost-accounting precursor and the fleet governance token view (PR #112 feature c); the coordinator's transcript dir path is already resolved by the watcher for limit detection — reuse that.
