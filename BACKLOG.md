this is the umbrella under which the six spun-out efficiency levers below sit. **Why (surfaced 2026-07-02 to 2026-07-03):** the sha-cache fix (a held PR re-ran the full healthcheck every ~90s tick and manufactured its own transient code-error→flaky on PR #65) is likely one symptom of a broader pattern — the whole pipeline probably repeats work and runs gates that are redundant, unnecessary for some diffs, or sequenced sub-optimally. **Scope — every stage end-to-end:** (1) backlog/scribe — per-item git pulls, drainer spawn/teardown churn; (2) lane spawn — worktree + SHARE_LINKS symlinks, multi-kilobyte argv prompts; (3) build — isolated builders re-deriving context from files; (4) WATCHER TICK LOOP (`agent-watch.sh`) — how many `gh`/`herdr`/`git` subprocesses it forks per tick (latency + rate-limit + cost), what it recomputes every tick vs could cache, and whether the tick cadence itself is right; (5) HEALTHCHECK gate — serialization (`HEALTH_CONCURRENCY=1`: always necessary, or can worktree-isolated suites parallelize?), retry-before-red, and the now-added sha-cache; (6) REVIEW gate — per-PR Opus cost, whether trivial/docs/test-only diffs need it at all (ties to the risk-tiered-review item), and whether review + healthcheck overlap; (7) auto-refix rounds + wake verification; (8) merge re-verification; (9) reap — worktree remove, tab close, backlog reap, orphan sweep. **Look for four things at each stage:** (a) REDUNDANT work (repeated subprocess calls, re-reads, re-runs of unchanged inputs); (b) SPEED-UPS (caching, parallelization, better sequencing, cheaper models/tools); (c) UNNECESSARY gates — anything duplicating GitHub's own branch-protection/required-checks or adding latency without safety value; (d) gate ISSUES / correctness — races, false-reds, the manufactured-transient class, gates that can cache a wrong verdict. **Method:** token/cost instrumentation is ✅ shipped (commit e2a0ae5 — `herd cost`; per-builder/review accounting); add latency instrumentation as the audit proceeds, then map the flow, then rank findings; each significant finding becomes its own item; validate any flow change on the sandbox-consumer sim rig before shipping. **Composes with:** the six efficiency levers below (its children), the herdkit-vs-harness EPIC in `## Someday / Deferred` (shares the instrumentation precursor), and the sandbox-consumer rig (its safe test bed). **DELIVERABLE SHIPPED:** `docs/flow-audit.md` landed as PR #77 (commit 14f7e42); - 🔜 **EPIC (deferred research): herdkit vs the raw Claude harness — falsify the null hypothesis that herdkit adds nothing a Workflow script can't already do** — **Null hypothesis (to test, not assume):** the raw Claude harness (Workflow tool + subagents + hooks) can already express herdkit's merge gating, worktree isolation, adversarial review, and parallelism — herdkit is just an opinionated Workflow script. The EPIC SUCCEEDS not by winning a within-session task battery (herdkit would LOSE that — more tokens, more failure surface, more orchestration complexity; running that benchmark would rig the result) but by finding ONE workload the raw harness ARCHITECTURALLY CANNOT complete. **The identified irreducible axis — TIME + PRESENCE decoupling:** the harness's execution and state are conversation-scoped by construction: a Workflow runs to completion and returns; subagents die with their parent; orchestration memory IS the conversation context — context exhaustion ends the run. Herdkit externalizes execution to detached OS processes and herdr panes, and externalizes state to git branches + PRs + the backlog file + the journal + lockfiles, so it survives session end, context exhaustion, machine restart, multi-day spans, and hand-off to a different operator. **Concrete success benchmark (the falsifying scenario):** drain a ~20-item backlog UNATTENDED overnight, surviving at least two usage-limit resets and a machine/terminal restart, with a hand-off to a SECOND operator mid-run — a scenario a single Workflow invocation structurally cannot complete. **Honest disposition clause:** if herdkit cannot win on the time/presence axis (e.g. the harness acquires durable execution primitives, or overnight unattended drain proves not to be a real use case), the EPIC's correct conclusion is to downgrade herdkit to 'an opinionated Workflow template' — NOT to declare victory because the within-session run was close; the study must be willing to reach that verdict and document it plainly. **Secondary benefit (weaker — note honestly):** opinionated installable packaging that routes every AI change through GitHub's OWN durable gates (branch protection + required checks) so every automated commit lands through the same auditable PR path a human's would; this is convention and convenience, not a capability moat, and erodes as the Workflow tool matures. **Litmus test:** finish before closing your laptop → use the raw harness; work must outlive your attention → herdkit. **Sequencing — unchanged:** (1) write a crisp positioning thesis grounded in the time/presence axis; (2) ✅ DONE — token/cost instrumentation: `scripts/herd/cost.sh` logs tokens + cost per builder and per review, surfaces "cost per merged PR" via `herd cost` (commit e2a0ae5); (3) ONLY THEN, gated behind the sandbox-consumer sim rig (existing `## Reliability / safety` item) + steps 1–2, run the time/presence falsification benchmark above — use a STUB builder mode (deterministic tiny commit, no model call) so the overnight run is fast and cheap. **Efficiency levers** (already spun out as individual items in `## Efficiency / cost`): ✅ REVIEW_ESCALATE_GLOB risk-tiered review (PR #73), ✅ externalize task specs (commit fd74659), ✅ SHA-cache the healthcheck (PR #66), ✅ token/cost instrumentation (commit e2a0ae5), prompt-cache-aware prompt ordering, evidence-based model escalation (a ✅ eco-starter-defaults PR #91; b/c/d ✅), ✅ match builder spawn rate to `REVIEW_CONCURRENCY` (PR #79).
