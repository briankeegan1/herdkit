## Reliability / safety

- 🚧 **Dead-builder detection — surface a spawned builder whose agent process vanished** *(worktree dead-builder-detection)* — detect the case where a worktree is present, no live agent exists in `herdr agent list`, and no open PR has been filed; emit a loud console row + notification so the operator knows the builder died silently. Motivated by the 2026-07-03 de-streamlit-docs builder silently dying (agent gone, pane destroyed, no PR, no commits) — found only when the operator noticed an empty tab. The watcher already handles (a) transcript-growth STALL for working-but-stuck builders (~line 1352), (b) `_sweep_orphan_tabs` for tabs with no worktree/PR (~line 749), (c) limit-hit auto-resume (~line 132) — the dead-pre-PR case falls through all of them. First increment: DETECT + loud alert only; auto-respawn deferred (edge cases: uncommitted work, respawn loops). Extends existing builder-liveness section without duplicating it. A broader control-room health sweep (watcher + backlog viewer + coordinator panes all alive) is a possible follow-up; cross-refs fleet EPIC's WATCHER HEARTBEAT MONITOR (feature f).
- 🔜 **COORDINATOR auto-resume + watchdog — close the session-scoped single point of failure that breaks unattended overnight runs** *(HIGH priority — linchpin for herdkit-vs-harness EPIC)* — **THE GAP (confirmed 2026-07-03):** builders get a `rate_limit` `StopFailure` hook (via `new-feature.sh` → `herd_write_ratelimit_hook`) and the watcher auto-resumes them at reset (`_detect_limit_hit` / `_resume_builder` / `.agent-watch-limit`). The coordinator (`coordinator.sh` → plain `claude --model MODEL_COORDINATOR /coordinator`) gets NO rate_limit hook, NO watchdog, NO auto-resume; the watcher self-excludes the coordinator tab and never touches it. So if the coordinator hits the account usage limit (or its process dies) unattended, it parks at Claude's interactive '1. Upgrade / 2. Stop and wait' menu and nothing revives it — self-resume paradox: a paused session can't resume itself; only a detached process can. **Impact:** in-flight work still completes (watcher merges ready PRs + resumes builders), but ALL orchestration requiring the coordinator (new spawn waves, human-verify approvals, backlog reaping, escalation handling) halts until a human resumes it — which breaks the 'unattended overnight, survive N limit resets' claim of the herdkit-vs-harness EPIC; the coordinator is the remaining session-scoped seam after execution + state were externalized. **Proposed design — two complementary pieces:** (1) **COORDINATOR-WATCHDOG** — a detached, non-LLM process (sibling to the watcher, or folded into `agent-watch.sh`, or a launchd/cron job) that detects the coordinator is limit-parked or dead (install the `rate_limit` hook on the coordinator like builders have; watch its `.herd-limit-sentinel` + `herdr agent list` status + the interactive menu) and at reset+buffer relaunches it via `claude --continue` in the coordinator pane — mirroring `_resume_builder`; must avoid double-launch (lock like the watcher's `WORKSPACE_NAME` launcher lock), bound retries, and handle the interactive-menu-parked case distinct from a clean session end; ties to the 'validate+harden limit-auto-resume through the interactive menu' item. (2) **EXTERNALIZED WORK QUEUE** — coordinator writes intended builder spawns into a durable queue file that the watcher drains (bounded by `REVIEW_CONCURRENCY`/`SPAWN_AHEAD`), so mechanical throughput survives a coordinator pause; design principle: push everything deterministic into the detached watcher so a coordinator pause barely matters. **Cross-refs:** fleet EPIC's WATCHER HEARTBEAT MONITOR (feature f) and GLOBAL CONCURRENCY/TOKEN GOVERNANCE (feature c); the limit-auto-resume item; the herdkit-vs-harness overnight benchmark (this is a PREREQUISITE for that benchmark to be trustworthy). **Note (2026-07-03):** the earlier live limit-hit did NOT cleanly validate builder auto-resume — a human interfered with the menu before auto-resume could act (see `limit-resume-menu-hazard` memory); both builder-menu-resume AND coordinator-resume need real unattended validation before the overnight benchmark is run.

## Efficiency / cost

> Spin-outs from the deferred [herdkit-vs-harness efficiency EPIC](#someday--deferred) (surfaced 2026-07-02). Token/cost instrumentation — `scripts/herd/cost.sh`, per-builder/review token accounting priced to USD, cost-per-merged-PR, `herd cost` command — is ✅ shipped (commit e2a0ae5); the instrumentation gate is lifted for all items below.


- ✅ **AUDIT (research pass): end-to-end flow redundancy, speed-ups, and gate necessity/correctness** — **Deliverable:** a findings doc (`docs/flow-audit.md`) + a RANKED list of concrete follow-up items, not a single code change; this is the umbrella under which the six spun-out efficiency levers below sit. **Why (surfaced 2026-07-02 to 2026-07-03):** the sha-cache fix (a held PR re-ran the full healthcheck every ~90s tick and manufactured its own transient code-error→flaky on PR #65) is likely one symptom of a broader pattern — the whole pipeline probably repeats work and runs gates that are redundant, unnecessary for some diffs, or sequenced sub-optimally. **Scope — every stage end-to-end:** (1) backlog/scribe — per-item git pulls, drainer spawn/teardown churn; (2) lane spawn — worktree + SHARE_LINKS symlinks, multi-kilobyte argv prompts; (3) build — isolated builders re-deriving context from files; (4) WATCHER TICK LOOP (`agent-watch.sh`) — how many `gh`/`herdr`/`git` subprocesses it forks per tick (latency + rate-limit + cost), what it recomputes every tick vs could cache, and whether the tick cadence itself is right; (5) HEALTHCHECK gate — serialization (`HEALTH_CONCURRENCY=1`: always necessary, or can worktree-isolated suites parallelize?), retry-before-red, and the now-added sha-cache; (6) REVIEW gate — per-PR Opus cost, whether trivial/docs/test-only diffs need it at all (ties to the risk-tiered-review item), and whether review + healthcheck overlap; (7) auto-refix rounds + wake verification; (8) merge re-verification; (9) reap — worktree remove, tab close, backlog reap, orphan sweep. **Look for four things at each stage:** (a) REDUNDANT work (repeated subprocess calls, re-reads, re-runs of unchanged inputs); (b) SPEED-UPS (caching, parallelization, better sequencing, cheaper models/tools); (c) UNNECESSARY gates — anything duplicating GitHub's own branch-protection/required-checks or adding latency without safety value; (d) gate ISSUES / correctness — races, false-reds, the manufactured-transient class, gates that can cache a wrong verdict. **Method:** token/cost instrumentation is ✅ shipped (commit e2a0ae5 — `herd cost`; per-builder/review accounting); add latency instrumentation as the audit proceeds, then map the flow, then rank findings; each significant finding becomes its own item; validate any flow change on the sandbox-consumer sim rig before shipping. **Composes with:** the six efficiency levers below (its children), the herdkit-vs-harness EPIC in `## Someday / Deferred` (shares the instrumentation precursor), and the sandbox-consumer rig (its safe test bed). **DELIVERABLE SHIPPED:** `docs/flow-audit.md` landed as PR #77 (commit 14f7e42); ranked follow-up items from that doc still need triaging into individual backlog items.


## Someday / Deferred

- 🔜 **EPIC (optional, enterprise-oriented): Master / fleet coordinator — a parent orchestration layer above the per-project coordinators** *(✅ P0 shipped PR #94 — project registry + herd fleet register/list/status/upgrade/reload; ✅ P1 digest shipped PR #99; ✅ P2 inbox shipped PR #102; P3 NL agent 🔜, P4 governance ✅ shipped PR #112, keep EPIC open)* — **Vision (2026-07-02):** manage N herd projects (e.g. herdkit + northstar) from one meta-workspace — fan a single command out to ALL repos (e.g. `herd upgrade` into every repo at once), dispatch a task down to one or many projects, route natural-language commands ("send this to herdkit", "what is going on with northstar today?"), and get a cross-project daily digest of everything done across projects; the master knows the operator's full workflow and the relationships between projects; the operator can still hone in and work in one individual project's control room as needed. **Why it fits:** this is the clearest differentiator herdkit has over the raw Claude harness — persistent, multi-project, human-supervised across days; the harness is single-session/single-repo and architecturally cannot do this; cross-reference the herdkit-vs-harness EPIC — the master coordinator is a flagship instance of herdkit's only real moat (decoupling from session lifetime + operator presence, now across many projects). **Architecture sketch:** a coordinator-of-coordinators in its own meta-workspace over (1) a PROJECT REGISTRY mapping each project's `WORKSPACE_NAME` → `path/repo/.herd/config` (explicit `~/.herd/fleet` list or auto-discovered by scanning roots for `.herd/config`; `herdr workspace list` already enumerates live ones); (2) TWO COMMAND CHANNELS — a DETERMINISTIC fan-out (no LLM) that loops the registry running existing per-project herd commands (`update/reload/status`) and returns a per-project rollup table (the upgrade guard already refuses on dirty tree / mid-flight builders, making bulk-upgrade safe), and an NL ROUTING channel that resolves target label(s) and dispatches a task into each target project's coordinator (via `herdr pane run`, the same nudge mechanism) or a per-project inbox; (3) JOURNAL AGGREGATION for status/digest — every project already writes `.herd/journal.jsonl` (`herd why/log`), so the master rolls those up across projects. **Additional features:** (a) CROSS-PROJECT ATTENTION INBOX — one view of what needs the human right now across all projects (blocked PRs, human-verify holds, conflicts, failed gates); (b) DAILY DIGEST / STANDUP built from aggregated journals (shipped/blocked/in-flight per repo); (c) GLOBAL CONCURRENCY + TOKEN GOVERNANCE — cap total builders/reviews across all projects, since the Claude account usage limit is account-wide (observed 2026-07-02: herdkit + northstar competing for one quota caused a mid-task limit hit); this is the right place to prevent limit-hits and ties into the efficiency EPIC's cost instrumentation and the limit-auto-resume item; (d) POLICY PROPAGATION — set `MERGE_POLICY` / model tier / `TOKEN_MODE` across the whole fleet at once; (e) DEPENDENCY-AWARE UPGRADE FAN-OUT — when herdkit ships a breaking engine change, auto-queue the upgrade in consumer repos (ties into `.herd/deps` + the herd-upgrade versioned-migrations item); (f) WATCHER HEARTBEAT MONITOR — per-project "is each watcher alive", now attributable via the `herd-watch-<workspace>` argv0 marker shipped in PR #65; (g) register/deregister a project with the fleet. **Dependencies:** HARD dep on the workspace-scoping isolation hardened in argv0 attribution PR #65 and the launch-binding-guard in-flight item — safe fan-out and per-project liveness attribution rest on it; builds on the existing cross-repo link registry (`.herd/links`) + `.herd/deps` foundation; SOFT dep on the sandbox-consumer sim rig as the safe multi-project test bed (fan-out to N fake projects, assert isolation + rollup) — not required to design it, and P0 is testable carefully against real projects. **Phasing:** P0 ✅ shipped PR #94 — deterministic `herd fleet register/list/status/upgrade/reload` + project registry; P1 ✅ shipped PR #99 — cross-project digest/standup; P2 ✅ attention inbox shipped PR #102; P3 🔜 the NL master-coordinator agent; P4 ✅ shipped PR #112 — relationship graph + policy propagation + global concurrency/token governance. **Smoke-tested live (2026-07-03):** herd fleet list/status/digest handle an empty registry gracefully. **Relevant existing primitives:** `.herd/links`, `.herd/deps`, `herd report --to`, `journal.sh`, `herdr workspace list`, per-project `WORKSPACE_NAME` resolution. **Competitive landscape:** see the shared _Competitive Landscape & Prior-Art Research_ item below — run a live, dated `/deep-research` pass (NOT answered from model training memory) before committing to architecture;
