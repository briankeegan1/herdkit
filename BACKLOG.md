## Workflow control & discoverability

> Build order: F1 (merge policy) first; capabilities manifest + template-sync guard as one PR; init detection + flow-preference interview as one PR; herd config last (leans on the manifest for key validation).

- 🔜 **Init-time GitHub detection** — `cmd_init` (bin/herd:104-284) makes zero `gh api` calls today; add a detection pass: branch protection on the default branch, required-review count, required checks, CODEOWNERS, allowed merge methods; show findings and derive safe defaults (required reviewers present ⇒ default `MERGE_POLICY=approve`, never `auto`); degrade gracefully without remote/gh.
- 🔜 **Flow-preference interview + draft-PR flow** — New `herd init` questions → config keys: `PR_FLOW=direct|draft` (draft: lanes instruct builders `gh pr create --draft`; watcher already holds `DRAFT`, agent-watch.sh:157), `PR_READY_WHEN=builder|coordinator|human`, `LOCAL_REVIEW=none|pre-pr` (run the review gate in the worktree BEFORE the PR is public vs today's post-PR review), `MERGE_METHOD`, `DELETE_BRANCH_ON_MERGE`; thread prefs into the lane rules text (herd-quick.sh:58-61, herd-feature.sh:50-53).
- 🔜 **`herd config` + coordinator Workflow settings mode** — `herd config list|get|set` with key validation, aware of what each change requires: watcher keys ⇒ restart watcher (pid file `.watcher-<slug>.pid`), coordinator-facing keys ⇒ re-render skill (render step at bin/herd:277-278); coordinator menu gains a 'Workflow settings' entry to view/change any workflow pref anytime and relaunch affected pieces — nothing is init-only.
- 🚧 **`herd init` must end in a verified 3-pane control room** *(worktree: utf8-control-room — tackling the issue #31 cp1252 root-cause slice: UTF-8-safe python3 subcalls + loud layout failures; init-launch and self-heal remain 🔜)* — first external-consumer test (friend's project 'bootyslime', Windows) ended with ONLY a coordinator pane: no backlog-view pane, no herd-watch pane, no watcher process — and the rendered coordinator skill then misexplained the design to the user, claiming the watcher pane only appears once a PR exists (false: it is persistent from launch). Root causes: (1) `cmd_init` never launches the control room — it only PRINTS a next-step hint (`bin/herd:313`) pointing at `coordinator.sh`, easy to miss; (2) `coordinator.sh` fails silently and partially — every `herdr` call discards output (`>/dev/null`) and the script runs under `set -euo pipefail`, so a failed backlog-view 'pane run' (`coordinator.sh:58`) or watch-pane split (`coordinator.sh:68-72`) aborts mid-layout with NO error message, leaving a lone coordinator pane (the friend's tab label matched `HERD_TAB_COORDINATOR`, proving `coordinator.sh` ran and died partway); (3) the coordinator template has no control-room contract, so a coordinator waking in a broken layout can't detect or self-heal it. **Confirmed root cause (issue #31, Windows/bootyslime):** python3 defaults stdout/stderr to the cp1252 codepage on Windows, and the herd scripts' `python3 -c` subcalls parse/print strings containing emoji/non-ASCII (🐑/🛰/✍️ pane labels), so the subcall throws a `UnicodeEncodeError`; under `coordinator.sh`'s `set -euo pipefail` + `>/dev/null` this error is silently swallowed and the layout truncates mid-run to a lone coordinator pane — this is the concrete failure mode that explains the observed symptom. Fix: `export PYTHONUTF8=1` (or `PYTHONIOENCODING=utf-8`) once in `scripts/herd/herd-config.sh` so every `python3` subcall across the engine is UTF-8-safe regardless of platform codepage; also keep non-ASCII out of machine-parsed output paths. Build: (a) `cmd_init` offers to launch `coordinator.sh` at the end (defaulting yes when `herdr` is available); (b) `coordinator.sh` verifies each step and reports exactly which pane failed and why instead of silently truncating the layout — including degrading gracefully when backlog-view deps (`glow`) are missing; (c) the rendered skill documents the expected layout (backlog-view | coordinator / watch below) and instructs the coordinator to check for the watch+backlog panes on invocation and re-run `coordinator.sh` to repair when missing; (d) sanity on Windows: `python3`-vs-`python`, codepage/encoding, and pane split/run availability belong in `herd-preflight.sh`. Related: 'Phase 4: onboard an external consumer' in Someday/Deferred — this is the first real finding from that test.
- 🚧 **`herd reload` — one-command workspace reload** *(worktree: herd-reload)* — no way today to bounce the herd control room: the watcher sources config ONCE at startup (agent-watch.sh:59, outside the poll loop), so a `.herd/config` change (e.g. the MERGE_POLICY auto flip on 2026-07-01) silently doesn't apply until someone hand-kills agent-watch.sh and relaunches herd-watch.sh; duplicate watchers also accumulate (two seen running concurrently on 2026-07-01 despite the spawn lock). Build a `herd reload` subcommand (+ capabilities.tsv row, per the new caps-sync guard): (1) stop ALL agent-watch.sh processes for this workspace (pgrep -f plus the `.watcher-<slug>.pid` lockfile — not just the pid-file one); (2) relaunch the watcher in its herdr pane (herdr pane run) so it re-reads config and runs freshly merged watcher code; (3) re-render the coordinator skill (`herd render`) so template/manifest changes land; (4) optionally recreate missing standard panes (coordinator / watch / scribe) in the workspace; (5) print what was reloaded and the effective MERGE_POLICY so the human can confirm the new state.
- 🔜 **Backlog-reconcile step in cutover / extraction / rename PRs** *(issue #17)* — when a PR moves or renames the things that backlog items point at (file paths, function names, section headers, worktree names), affected entries dangle silently after the PR lands. The coordinator / scribe should run a reconcile pass as part of landing such PRs: diff the rename surface, identify backlog entries that reference moved things, and update or flag them so the backlog stays coherent with the live codebase.
- 🔜 **Analyst seat launch config must include an explicit context-persistence tenet** *(issue #19)* — the playground analyst seat's herdkit-hosted launch prompt has no cross-session context-persistence instruction, so the seat can silently go stateless between sessions: the only instance of that behavior today survives as a pruneable memory entry in one consumer project (northstar), not as a durable directive in the seat config. Fix: add an explicit tenet to the analyst seat's launch prompt / config requiring it to actively build and update the user's project and domain context in the local project memory store on every session — context stays local and gitignored per the existing memory contract; this is a prompt/config-only change with no engine wiring.

## Reliability / safety

- 🔜 **Pre-trust fix (PR #22) is ineffective — builders still die on the folder-trust prompt** — new-feature.sh:43-49 writes an empty {} to `<worktree>/.claude/settings.json` claiming Claude Code treats any project-level settings file as a trust marker; that premise is wrong (folder trust is recorded in `~/.claude.json` keyed by project path), so freshly-spawned builders can still hit the interactive trust prompt and die/vanish with zero commits — observed with worktree watch-pr-display: agent pane died at startup leaving an empty shell pane, watcher correctly flagged 'idle · no PR'. Real fix candidates, to be verified against how Claude Code actually records trust: seed the project entry in `~/.claude.json` (e.g. `hasTrustDialogAccepted`) before launch, or have the launcher detect the trust prompt in the pane and answer it, or find a supported non-interactive trust flag. The stalled-builder detection half of PR #22 works and stays.
- 🔜 **Stall detector: check worktree mtimes before flagging** — `agent-watch.sh:383-392` flags any working builder with zero branch commits after 5 minutes as 'stalled? · check pane', but builders normally commit ONCE at the end right before `gh pr create`, so every normal build >5 min shows a false stall warning (observed 2026-07-01: herd-reload and workspace-pin both flagged while actively editing — one had file edits 0 minutes old). Refine: before flagging, check the newest mtime among the worktree's dirty/untracked files (`git status --porcelain` paths); recent edits (< ~5 min) ⇒ show 'building (uncommitted changes)' instead of the stall warning; only flag when the tree is clean AND commitless AND old, or dirty but untouched for >N min. Keep the warning for the genuinely-dead case (agent status not 'working' already shows 'idle · no PR').
- 🔜 **Mandatory dependency checks in install.sh + a `herd doctor` preflight** — the first external-consumer install (Windows) produced a partially-broken control room with no error pointing at the cause; a missing load-bearing dependency is the prime suspect, and nothing checks them today: `install.sh` verifies zero dependencies, `cmd_init` only soft-probes `claude` (bin/herd:149) and `gh` (bin/herd:189), and `herd-preflight.sh` checks ONLY herdr CLI presence + JSON contract. Notably `python3` — used by nearly every lane script to parse herdr JSON (`python3 -c` in coordinator.sh, scribe.sh, herd-quick.sh, herd-feature.sh, herd-review.sh, research.sh, herd-resolve.sh) — is never checked anywhere and is a classic Windows gap (often only `python` exists). **Confirmed root cause (issue #31, Windows/bootyslime):** beyond mere presence, python3 on Windows defaults stdout/stderr to the cp1252 codepage, and the herd scripts print strings with emoji/non-ASCII (🐑/🛰/✍️ pane labels) through `python3 -c`, causing a `UnicodeEncodeError` that kills the subcall; `coordinator.sh`'s `set -euo pipefail` + `>/dev/null` silently swallows the error and the layout truncates to a lone coordinator pane — this is the actual failure mode. Fix: `export PYTHONUTF8=1` (or `PYTHONIOENCODING=utf-8`) once in `scripts/herd/herd-config.sh` so every `python3` subcall is UTF-8-safe regardless of platform codepage; keep non-ASCII out of machine-parsed output paths. The doctor must check not just python3 presence but also its effective codepage/encoding on Windows. Build: (1) extend `herd_preflight` (scripts/herd/herd-preflight.sh) into a full dependency doctor — HARD-require `git`, `gh` (+ `gh auth status`), `claude`, `python3`, and `herdr`; report every missing dep in one pass with per-platform install hints, not fail-on-first; (2) run it from `install.sh` and at the top of `cmd_init` so a broken environment is caught before any config is written; (3) optional deps stay soft with explicit fallback notes: `glow` (backlog-view.sh:63-64 already degrades), `shellcheck`/`bats` (healthcheck already degrades); (4) expose as `herd doctor` so a user with a weird setup can self-diagnose anytime. Related: the '`herd init` must end in a verified 3-pane control room' item — doctor is the first gate of that verified launch.
- 🔜 **Healthcheck must gate / flag interaction tests for app-facing PRs** *(issue #15)* — the consumer healthcheck profile today runs only a render smoke (e.g. `st.testing.v1.AppTest.from_file`), which is blind to broken interactivity: a persistent quick-set radio silently overrode a slider every re-run in northstar PR #48 while the smoke passed. Engine-side ask: the lane rules / healthcheck contract should let a project declare interaction-test expectations (e.g. Streamlit `AppTest`: drive the widget, then assert the dependent output changes) and at minimum flag their *absence* for PRs that touch the app surface — a healthcheck that cannot observe widget → output causality should not silently green-light an app-facing PR.
- 🔜 **Review gate must distinguish infra death from a refused verdict** — observed on PR #32 (2026-07-02): the watcher's review capture (agent-watch.sh:487 command substitution → herd-review.sh → claude -p) was killed mid-review (prime suspect: a builder verifying kill logic whose pgrep -f agent-watch.sh sweep matched the watcher's own review fork), the capture returned EMPTY, and the gate posted 'no parseable verdict (defaulting to BLOCK)' and CACHED that BLOCK against the sha — so a purely infrastructural death became a sticky verdict requiring manual ledger surgery, while the orphaned reviewer finished with no pipe to report through (no PR comment, no verdict recorded). Build: (1) treat an EMPTY capture (vs an explicit 'REVIEW: BLOCK — reason') as INFRA-FAIL: do not cache, retry next poll with a retry cap, and say 'review infrastructure failed' not 'reviewer blocked'; (2) herd-review.sh should trap SIGTERM/SIGPIPE and flush 'REVIEW: INFRA-FAIL' so a killed run still reports; (3) make the reviewer chain resistant to stray-sweeps: distinctive process naming (e.g. exec -a herd-review-gate) so kill-by-pattern tools can exclude it, and lane rules must forbid builders from running kill/stop logic against the live workspace during verification (third live-infra interference tonight: killed the production watcher, leaked five strays, now severed an in-flight review).
